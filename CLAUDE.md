# ait-transit プロジェクト方針

## 目的
愛工大生向けの時刻表・乗継アプリ。
収益化（広告等）を視野に入れた個人開発で、
就活ポートフォリオも兼ねる。
最優先は「データの正確さ」。
嘘の時刻を1件でも出したら信頼を失うアプリである。

## ターゲットと優先ユースケース
対象路線: 学内シャトルバス / リニモ / 愛知環状鉄道
（八草駅はリニモ・愛環の両方が乗り入れる結節点）

1. 大学→帰宅方向の次の乗継
   （リニモ藤が丘方面・愛環高蔵寺方面・愛環岡崎方面）
2. 沿線→大学（通学）の次の乗継（同上の逆方向）
3. シャトルバスの次の便

## データ品質ルール（最重要）
- 時刻表データは絶対に記憶から生成しない。
  必ず公式ソースを取得してパースする
  （リニモ: linimo.jp / シャトル: ait.ac.jp）
- データ追加・修正後は必ず実施して報告:
  - 独立再パースでの全件照合
  - ランダム5件の公式照合
  - 機能テスト
  - tsc --noEmit
- 既存データの変更は差分を明示する

## レビュー方針
- 大きめの変更はCodexにコードレビューを依頼
  （利用上限中は自己レビューで代替し、その旨記録）

## コマンド
- 開発: npm run dev
- ビルド確認: npm run build
- 型チェック: tsc --noEmit
- テスト: npm test（回帰テスト。データ変更後は必ず実行）
- 動作確認: /?test_date=2025-10-15&test_time=09:00:00
  で任意日時をシミュレート可能

## 構成の要点
- src/data/*.json: 時刻表データ（DBなし、JSON直管理）
- src/lib/timetable.ts: 乗継計算ロジック全部
- src/components/MainClient.tsx: メインUI

## データ構造の罠（必読）
- 終着駅では「到着時刻」が進行方向ラベルで
  記録されている（例: 八草のto_yagusaは到着時刻）。
  発車時刻と混同しないこと
- day_type: weekday_green=平日 / holiday_red=土休日。
  公式表のオレンジ時刻=土休日・学校休業期間
- 時刻フォーマットはゼロ埋めなし混在（"5:31:00"等）。
  比較は必ずtimeToMinutes経由で行う
- 深夜0時台のエントリ（0:07:00等）が存在する。
  日付またぎの扱いに注意
- 愛環は黒時刻=毎日運転（両エントリ）、赤時刻=土休日運休
  （weekday_greenのみ）。あさシャトル（青時刻・新豊田⇔三河豊田、
  運転日変動）は八草に到達しないため収録対象外
- 愛環のterminal（行先）は到達可否判定に使用。手前止まり
  （北野桝塚行き等）は目的駅に届かないので除外される
